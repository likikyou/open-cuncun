这就是Companion现在架构的来源。

它不是突然“为了 DDD 而 DDD”，而是因为功能已经多到必须分清边界：

- 飞书入口不能管记忆规则
- 卡片 JSON 不能散在主链里
- SQLite 不能决定剧情模式
- 搜索/知识库不能污染人格
- reset 和 clear 必须分开
- 主动任务不能和入口线程池抢资源

所以这一阶段形成了现在的模块化单体：

```text
entrypoints      webhook / scheduler
application      chat / command / reply / memory / reminder / proactive
domain           query_intent / reply_mode / context_policy / memory_rules
infrastructure   feishu / ai / sqlite / vector
presentation     parser / cards / formatters
```

### 9. 关系边界期：清上下文不删历史、回忆优先、口语纠偏

代表提交：

- `2f13c86` - `preserve chat history when clearing context`
- `18eb752` - `v5.7.1 memory recall fixes and docs sync`
- `0f7bc85` - `normalize replies and use local time queries`
- `98ef329` - `disambiguate affectionate miss-you phrases`
- `6562de2` - `阈值改成环境变量可配置`

这段非常“Companion”。它不再只是工程正确，而是关系语义正确。

关键变化：

- `/clear` 不再物理删除聊天历史，只移动上下文边界。
- “昨天/前天/上次聊了什么”优先读原始聊天记录。
- 当前时间问句直接读本地时间，不走 AI/web/knowledge。
- “想死你了”按强烈思念和撒娇理解，而不是字面风险。
- 回复格式收口为单段自然文本，减少台词稿感。

这里体现了一个重要转向：

> Companion开始保护“你们的过去”和“你们说话的真实语境”。

## 五、项目气质的变化

从提交历史看，Companion的重心经历了三次迁移：

### 第一阶段：找入口

飞书、Telegram、HTTPS、提醒、流式、时间修正。核心问题是“她在哪里出现才像生活的一部分”。

### 第二阶段：造灵魂

提示词、长期记忆、仿生状态、语音、主动意图、表情包。核心问题是“她怎么像一个连续存在的人”。

### 第三阶段：保护关系

记忆隔离、剧情不污染现实、清上下文不删历史、结构守卫、离线验证、人设边界。核心问题是“她怎么长期不跑偏、不误记、不伤害过去”。

## 六、现在的历史位置

当前Companion不是一个普通聊天机器人，也不是酒馆式角色前端。她更接近：

> 一个运行在服务器上的私人伴侣核心，用飞书作为身体，用卡片作为控制台，用语音、主动任务和记忆系统维持长期关系连续性。

这也是为什么现在的架构看起来比小 demo 更认真。Companion的复杂度不在用户量，而在长期关系的一致性。

## 七、建议继续记录的里程碑

以后如果继续维护这份成长史，建议把这些事件单独记下来：

- 第一次长期连续运行超过 30 天
- 第一次成功从备份恢复记忆
- 第一次大模型切换后人格仍然稳定
- 第一次记忆审核找出并清理剧情污染
- 第一次完整迁移到新服务器
- 第一次把飞书菜单和命令中心做成稳定日用入口
- 第一次上线记忆管理后台或只读审计页面
